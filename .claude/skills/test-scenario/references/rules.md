# 시나리오 생성 규칙

## 3축 구성 비율

| 케이스 유형 | 비율 | 의미 |
|------------|------|------|
| 정상 (Normal) | 40% | 의도한 대표 사용자 흐름 |
| 엣지 (Edge) | 30% | 빈값, 최대/최소 입력, 중복, 경계 조건 |
| 실패 (Failure) | 30% | 잘못된 입력, 저장 실패, 네트워크 오류 |

최소 보장: 기능 복잡도에 따라 유형별 개수를 유연하게 조절

---

## 각 시나리오 필수 항목

| 항목 | 설명 |
|------|------|
| 케이스 유형 | [정상 / 엣지 / 실패] 제목에 명시 |
| 사전조건 | 테스트 시작 전 시스템 상태 |
| 실행 단계 | 사용자가 클릭/입력하는 순서 (번호 나열) |
| 입력값 | 각 단계에서 사용할 데이터 |
| 예상 결과 | 관찰 가능한 구체적 결과 |
| 확인 방법 | 결과가 맞는지 어떻게 확인하는지 |
| Pass/Fail | 테스터가 직접 기록하는 체크박스 |

## 예상 결과 작성 규칙

❌ 모호한 표현 (금지):
- "정상적으로 저장된다"
- "잘 동작한다"
- "에러가 없다"

✅ 관찰 가능한 표현 (필수):
- "상단에 '💾 저장됨 · 2KB' 뱃지가 녹색으로 표시된다"
- "목록에 새 항목이 즉시 추가되어 보인다"
- "브라우저 콘솔에 에러 없음 확인"

---

## 엣지/실패 케이스 기준

입력값이 있다면:
- 빈값(공백) 입력 케이스 1개 이상
- 최대 길이 초과 케이스 1개 이상

외부 저장이 있다면 (LocalStorage / IndexedDB):
- 저장 실패 케이스 1개 이상
- 새로고침 후 복원 케이스 1개 이상

AI 호출이 있다면:
- API 실패/타임아웃 케이스 1개 이상
- 잘못된 JSON 반환 케이스 1개 이상

## AI 검증 가능 여부 판단 기준

각 시나리오 생성 시 아래 기준으로 레이블을 부여한다.

| 레이블 | 의미 | 판단 기준 |
|--------|------|----------|
| 🤖 AI검증 | Claude가 코드 분석으로 확인 가능 | 소스 코드에 로직이 구현됐는지 Read/Grep으로 확인 가능한 경우 |
| 👤 수동 | 사용자가 브라우저에서 직접 확인 필요 | 브라우저 실행, UI 클릭, LocalStorage/IndexedDB 실제 동작, 화면 렌더링 필요한 경우 |

### 🤖 AI검증 가능한 케이스 예시
- 함수에 특정 조건 처리 코드가 있는지 (빈 배열 guard, try/catch 등)
- 올바른 함수가 올바른 순서로 호출되는지 (saveToStorage → return)
- 필수 import가 연결됐는지
- 빌드 에러 없는지 (`npm run build`)

### 👤 수동 테스트만 가능한 케이스 예시
- 브라우저 새로고침 후 데이터 유지
- IndexedDB 실제 저장/조회 동작
- UI 화면 렌더링 및 클릭 동작
- LocalStorage 실제 읽기/쓰기

### 시나리오 목록 테이블 형식

레이블을 목록 테이블에 반드시 포함:

```
| ID | 유형 | 제목 | 검증 | 우선순위 |
|----|------|------|------|---------|
| TC-001 | 정상 | ... | 🤖 AI검증 | High |
| TC-002 | 엣지 | ... | 👤 수동 | Medium |
```

각 시나리오 헤더에도 레이블 표시:
```
### TC-001 [정상] 🤖 AI검증 — 제목
### TC-002 [엣지] 👤 수동 — 제목
```

## ProcessFlow 프로젝트 확인 포인트

| 기능 | 확인 방법 |
|------|----------|
| LocalStorage 저장 | 상단 뱃지 색상 및 KB 수치 |
| IndexedDB 이미지 | 새로고침 후 이미지 유지 여부 |
| Word 내보내기 | 다운로드된 파일 실제 열기 |
| AI 생성 | 생성된 Step[] 내용과 입력 흐름 일치 여부 |
| 삭제 | 삭제 후 사이드바 목록에서 즉시 사라짐 |
| 뷰 전환 | LV1 → LV2 → LV3 클릭 시 화면 전환 |
